로우코드 툴, 아니면 바이브 코딩?

오리지널 당라이브 Danglive
2025년 05월 05일 14:32 상하이
사람들은 항상 게으름을 피우려고 한다. 코드를 작성하지 않고도 드래그 앤 드롭으로 앱을 만들 수 있다면, 누가 코드와 씨름하고 싶어 하겠는가? 하지만 정말로 로우 코드 플랫폼으로 완벽하게 작업할 수 있을까? 아니면 그저 입문용 장난감에 불과할까?
반년 전으로 돌아가자면: ESP32 개발을 처음 배우기 시작했을 때, Micropython 코드를 배우기 귀찮아서 무슨 로우 코드 도구가 있으면 좋겠다고 생각했다. 즉, WYSIWYG 방식으로 몇 번 끌어다 놓고, 몇 가지 매개변수만 입력하면 프로그램이 완성되는 그런 도구 말이다. 그러나 뜻대로 되지 않아서, 국내외의 몇 가지 로우 코드 도구를 시도해 본 후에야 결국에는 Pycharm과 Thonny로 돌아가서 개발하는 것이 가장 효율적이라는 걸 깨달았다.
우선, 로우 코드 도구 자체가 블랙박스입니다. 컴파일된 코드의 가독성이 매우 떨어져서 어떻게 작동하는지 알 수 없습니다. 기능이 요구사항을 충족하지 못할 경우 디버깅이 거의 불가능하며, 문제 해결은 추측에 의존할 수밖에 없습니다. 예를 들어 시뮬레이션 환경에서는 정상적으로 작동하는 기능이 실제 SoC에서 실행되면 동작하지 않을 때, 문제의 원인을 찾아내기 매우 어렵습니다.
둘째, 사례를 통해서만 배울 수 있지만 사례의 수가 너무 적을 수 있어 원하는 기능 구현을 위한 사례가 없으면 여기저기 찾아다녀야 하고 검색하는 데 많은 시간과 노력이 소요됩니다.
그래서 1, 2주 동안 해보다가 다시 pycharm으로 돌아가기로 결정했고, 통의링마를 AI 프로그래밍 어시스턴트로 사용했는데, 효율이 사실 매우 높았어요. 통의링마는 IoT 코드에 대해 최적화되었다고 해서 당시에 그것을 선택했어요.
물론 AI 프로그래밍 도움말을 사용하더라도 기본적인 프로그래밍 기초는 필요합니다. 그렇지 않으면 AI가 제공한 코드의 문제점을 식별할 수 없기 때문입니다. 종종 AI 도움말이 코드를 제공하는 데는 2분밖에 걸리지 않지만, 특정 문제를 디버깅하는 데는 이틀이 걸릴 수도 있습니다.
Vibe 코딩은 로우코드 도구를 사용하는 것보다 두 가지 핵심 문제에서 확실한 이점이 있습니다. 첫째는 디버깅입니다. 문제가 발생하면 직접 코드를 읽거나 오류 메시지를 AI 어시스턴트에 계속 전달하여 해결할 수 있어 제어 가능성이 크게 향상됩니다. 둘째는 더 이상 검색할 필요가 없다는 점입니다. 요구 사항을 AI에 직접 알리면 지속적으로 코드를 제공할 수 있으며, 이전에 찾아야 했던 일부 라이브러리도 직접 작성할 수 있습니다.
예를 들어 반년 전에 7-세그먼트 디스플레이의 드라이버 프로그램이 필요했는데, Github의 라이브러리들은 모두 마음에 들지 않았습니다. 해당 모델의 7-세그먼트 디스플레이에 대한 공통 캐소드 연결 방식의 드라이버 프로그램만 있었고, C 언어 버전뿐이었습니다. 하지만 제 것은 공통 애노드 방식의 7-세그먼트 디스플레이였고, Micropython 언어 버전이 필요했습니다. AI 도우미를 사용해 이 C 언어 버전의 드라이버 코드를 Python 언어 버전으로 다시 작성하는 것은 매우 편리했습니다. 다만 공통 캐소드에서 공통 애노드로 변경하는 부분은 AI 도우미가 이해하지 못해, 그 부분은 결국 제가 직접 논리를 수정해 완성했습니다. 만약 전체 프로젝트가 로우 코드 플랫폼으로 이루어져 있었다면, 손쓸 방법이 없었을 것입니다. 기성 라이브러리나 드라이버 프로그램을 찾을 수 없다면, 하드웨어를 교체하는 수밖에 없었을 테니까요.
최근 로컬에서 LLM을 배포하려고 시도하면서 비슷한 문제를 또 마주쳤다. 처음에는 홍색책(小红书)에서 dify, coze, n8n을 추천하는 다양한 글에 끌려 B站에서 공부를 시작했다. dify에 대한 문제는 docker 배포부터 고민이 되기 시작했다. 내 windows의 WSL 환경에서 docker 배포에는 항상 문제가 있었는데, 아마도 windows의 프록시를 활용할 수 없어서 docker 안에서 배포할 때 서버에 연결되지 않았고, 결국 포기했다. 그런데 모두 n8n이 더 성숙하다고 해서 살펴보니 확실히 그랬고, 그래서 n8n으로 시도를 바꿨다.
처음에는 정말 완벽해 보였어요. n8n에는 수백 가지의 즉시 사용 가능한 컴포넌트 통합이 있었고, AI 에이전트와 연결할 수 있으며 로컬에 ollama로 배포한 LLM을 호출하는 것도 매우 편리했어요. 간단한 테스트를 몇 가지 해보니 정말 좋았죠. 하지만 깊이 들어가면서... 문제가 나타나기 시작했어요. 실제 워크플로우에서 로컬 파일을 읽어야 할 때, 그것은 알 수 없는 오류를 내기 시작했어요. 'read file' 컴포넌트를 추가했더니 date-fns 오류가 발생하기 시작했어요. 오류 메시지를 AI에게 던져주니 date-fns의 최신 버전을 글로벌로 설치하라고 조언했어요. 좋아요, 그렇게 했죠! 이제 괜찮아진 것 같나요? 그런 다음 'extract from file' 컴포넌트를 추가했더니 또 오류가 발생했어요. 이번에는 완전히 길을 잃었고, 완전히 비극이 되었어요. 여러 번 재설치하고 공식 웹사이트에서 예제 워크플로우를 다운로드해도 같은 오류가 반복되었어요.
결각 깨달았다. 그냥 성실하게 VSCODE로 돌아가서 AI 도우미의 도움을 받아 프로그래밍하는 게 낫다. 하룻밤 만에 필요한 작은 도구도 완성되었다. 모든 '접착제' 코드는 AI가 거의 완벽하게 구현해 준다. 하지만 나는 비교적 보수적인 편이라 보통 통의링마의 '지능형 Q&A' 기능만 사용한다. AI가 제공한 코드는 보통 내가 직접 검토한 후에 복사하고 붙여넣기 한다. '파일 편집'과 '에이전트' 두 가지 기능 옵션이 효율적일 수 있지만, 코드를 엉망으로 만들까 봐 오히려 득보다 실이 클까 걱정이다.
저코드 플랫폼을 살펴보면 실제로 사용 문턱이 그리 낮지 않습니다. n8n과 같은 도구들은 각 컴포넌트마다 여전히 몇 가지 파라미터를 설정해야 하고, JS의 기본 문법을 이해해야 하며, JSON에 대해 알 필요가 있습니다. 오류가 발생하면 설치 환경에서의 문제를 해결하는 방법도 알아야 합니다. 이러한 컴퓨터 기본 지식은 어떤 도구나 플랫폼을 사용하든 결국 배워야 하는 부분입니다.
그럼 로우 코드 플랫폼은 사용할 수 없나요?
내 대답은: 쓸 수 있고, 여전히 매우 유용합니다! 첫째, 입문할 때 플로우차트가 작업 흐름 조직에 대한 직관적인 인식을 제공합니다. 큰 작업을 어떻게 작은 구성 요소로 분해하는지, 이러한 도구들은 이미 매우 일반화된 추상화를 해놓아서 직접 고민할 필요가 없습니다. 이러한 추상화는 vibe coding할 때 사고를 정리하는 데 큰 도움이 됩니다. 둘째, 수많은 사례들은 학습자가 자신에게 맞는 응용 시나리오를 창조하도록 영감을 줍니다. 자신의 요구에 맞춰 어떤 작업을 자동화함으로써 효율을 크게 향상시킬 수 있는지 볼 수 있습니다. LLM이라는 "망치"가 있는데, 다른 사람들이 어떤 "못"을 박았는지 보는 것이 망치를 들고 여기저기 못을 찾는 것보다 약간 덜 수고롭습니다.
마지막으로 느낀 점은 현재 가장 효과적인 도구는 여전히 vscode나 pycharm에 Cline, 통의링마 같은 AI 도우미를 결합한 것이며, 가장 가치 있는 직원은 여전히 업무 경험과 사고방식을 갖추고 컴퓨터 전문 지식 배경을 가진 올라운드 플레이어라는 것입니다.

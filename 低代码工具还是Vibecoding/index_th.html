เครื่องมือ Low-code หรือ Vibe Coding?

ต้นฉบับ Danglive Danglive
5 พฤษภาคม 2025 14:32 เซี่ยงไฮ้
มนุษย์มักจะขี้เกียจ โดยธรรมชาติ ใครจะอยากยุ่งกับโค้ด ถ้าสามารถสร้างแอปพลิเคชันได้แค่ลากแล้ววางโดยไม่ต้องเขียนโค้ด? แต่จริงๆแล้วแพลตฟอร์ม Low-code สามารถทำงานได้อย่างสมบูรณ์แบบหรือไม่? หรือเป็นแค่ของเล่นสำหรับผู้เริ่มต้น?
ย้อนกลับไปครึ่งปีก่อน: ตอนเริ่มเรียนรู้การพัฒนา ESP32 ขี้เกียจเรียนโค้ด micropython จึงหวังว่าจะมีเครื่องมือ low-code อะไรสักอย่างที่เห็นผลลัพธ์ทันที ลากวางสองสามครั้ง กรอกพารามิเตอร์ไม่กี่ตัวก็เสร็จโปรแกรม แต่ผลกลับไม่เป็นดังคาด หลังจากลองเครื่องมือ low-code ทั้งในและนอกประเทศแล้ว กลับพบว่ายังต้องกลับมาพัฒนาบน pycharm กับ thonny ถึงจะมีประสิทธิภาพสูงสุด
ประการแรก เครื่องมือ low-code นั้นเป็นกล่องดำในตัวของมันเอง โค้ดที่ถูกคอมไพล์ออกมามีความอ่านยาก รู้เพียงว่ามันทำงานแต่ไม่รู้ว่าทำงานอย่างไร เมื่อเกิดปัญหาที่ฟังก์ชันไม่เป็นไปตามที่ต้องการ การ debug ก็แทบจะเป็นไปไม่ได้ การแก้ปัญหาสามารถทำได้เพียงแค่คาดเดา ตัวอย่างเช่น ฟังก์ชันที่ทำงานปกติในสภาพแวดล้อมจำลอง แต่เมื่อมาทำงานบน SoC จริงกลับไม่ทำงาน ปัญหาอยู่ที่ไหนนั้นก็ยากที่จะค้นพบ
ประการที่สอง เรียนรู้ได้เฉพาะจากกรณีศึกษา แต่จำนวนกรณีศึกษาอาจมีน้อยเกินไป หากไม่มีกรณีศึกษาสำหรับฟังก์ชันการทำงานที่ต้องการ ก็จำเป็นต้องค้นหาทุกที่ ซึ่งใช้เวลานานและลำบาก
ดังนั้นหลังจากลองทำมา 1-2 สัปดาห์ ก็ตัดสินใจกลับมาใช้ pycharm โดยใช้ Tongyi Lingma เป็นผู้ช่วยเขียนโค้ด AI ซึ่งมีประสิทธิภาพสูงมาก Tongyi Lingma อ้างว่าได้รับการปรับปรุงสำหรับโค้ด IoT ดังนั้นตอนนั้นจึงเลือกใช้มัน
แน่นอนว่าการใช้ผู้ช่วยเขียนโปรแกรม AI คุณยังต้องมีพื้นฐานการเขียนโปรแกรมด้วยตัวเอง ไม่เช่นนั้นคุณจะไม่สามารถแยกแยะปัญหาต่างๆ ในโค้ดที่ AI ให้คุณได้ บ่อยครั้งที่ผู้ช่วย AI ใช้เวลาเพียง 2 นาทีในการให้โค้ดแก่คุณ แต่คุณอาจใช้เวลาถึง 2 วันในการแก้ไขข้อบกพร่องในปัญหาใดปัญหาหนึ่ง
Vibe coding เปรียบเทียบกับเครื่องมือ low code มีประโยชน์ที่เห็นได้ชัดในสองประเด็นสำคัญ ประการแรกคือการ debug เมื่อเกิดปัญหา คุณสามารถอ่านโค้ดด้วยตัวเอง หรือป้อนข้อความแสดงข้อผิดพลาดให้กับผู้ช่วย AI เพื่อแก้ไข ซึ่งเพิ่มความสามารถในการควบคุมได้อย่างมาก ประการที่สองคือไม่จำเป็นต้องค้นหาอีกต่อไป เพียงแค่บอกความต้องการให้ AI มันสามารถสร้างโค้ดให้อย่างต่อเนื่อง แม้แต่ไลบรารีบางอย่างที่เคยต้องไปค้นหาก็สามารถเขียนขึ้นมาเองได้
ตัวอย่างเช่น หกเดือนก่อน ต้องการไดรเวอร์สำหรับแสดงผลด้วยหลอดดิจิตอล แต่ไลบรารีบน Github ไม่ค่อยดีนัก มีเพียงไดรเวอร์สำหรับวิธีการเชื่อมต่อแคโทดร่วมของหลอดดิจิตอลรุ่นนั้นเท่านั้น และมีเฉพาะเวอร์ชันภาษา C ในขณะที่ของฉันเป็นหลอดดิจิตอลแบบแอโนดร่วม และต้องการเวอร์ชันภาษา micropython การใช้ AI ช่วยเขียนโค้ดไดรเวอร์จากเวอร์ชันภาษา C ใหม่เป็นภาษา python ทำได้ง่ายมาก แต่การเปลี่ยนจากแคโทดร่วมเป็นแอโนดร่วม AI ไม่เข้าใจในตอนนั้น ส่วนนี้สุดท้ายต้องแก้ไขลอจิกด้วยตนเอง หากโครงการทั้งหมดใช้แพลตฟอร์ม low-code ก็อาจจะหมดปัญญา ถ้าไม่พบไลบรารีหรือไดรเวอร์สำเร็จรูป ก็อาจต้องเปลี่ยนฮาร์ดแวร์แทน
เมื่อเร็ว ๆ นี้ลอง deploy LLM ในเครื่องอีกครั้ง แล้วก็เจอปัญหาแบบเดิมอีก ครั้งแรกถูกหลอกให้ลอง dify, coze, n8n จากบทความแนะนำมากมายใน Xiaohongshu เลยไปเริ่มศึกษาใน B站 ปัญหาเกี่ยวกับ dify เริ่มตั้งแต่การ deploy ด้วย docker ก็ปวดหัวแล้ว เพราะในสภาพแวดล้อม WSL บน windows ของฉัน การ deploy docker มักจะมีปัญหาบ้าง ส่วนใหญ่คือไม่สามารถใช้ proxy ที่อยู่ใต้ windows ได้ ตอน deploy ใน docker ก็เชื่อมต่อกับเซิร์ฟเวอร์ไม่ได้ สุดท้ายเลยยกเลิกไป แล้วทุกคนก็บอกว่า n8n นั้น mature กว่า ลองดูแล้วก็จริง เลยเปลี่ยนมาใช้ n8n ลองดูแทน
ตอนเริ่มต้น มันดูสมบูรณ์แบบจริงๆ นะ n8n มีส่วนประกอบที่รวมไว้แล้วหลายร้อยตัว สามารถเชื่อมต่อกับ AI Agent และการเรียกใช้ LLM ที่ติดตั้งในท้องถิ่นผ่าน ollama ก็สะดวกมาก ทำการทดลองง่ายๆ ก็ดี แต่เมื่อเจาะลึกเข้าไป... ปัญหาก็เริ่มปรากฏ: เมื่อฉันต้องการอ่านไฟล์ท้องถิ่นในเวิร์กโฟลว์จริง มันเริ่มแสดงข้อผิดพลาดโดยไม่ทราบสาเหตุ เพิ่มคอมโพเนนต์ "read file" แล้วเริ่มแสดงข้อผิดพลาดของ date-fns ยกข้อความผิดพลาดให้ AI มันแนะนำให้ติดตั้ง date-fns เวอร์ชันล่าสุดแบบ global ตกลง ทำตาม! ดูเหมือนจะดีขึ้น? แต่พอเพิ่มคอมโพเนนต์ "extract from file" ข้อผิดพลาดก็ยังคงอยู่ คราวนี้ไม่รู้จะแก้ไขอย่างไรแล้ว เป็นหายนะสนิท ติดตั้งใหม่หลายครั้ง และดาวน์โหลดเวิร์กโฟลว์ตัวอย่างจากเว็บไซต์ทางการ ก็ยังคงเจอข้อผิดพลาดเดิม
ในที่สุดก็ตื่นขึ้น เลือกกลับไปใช้ vscode อย่าง老老实实 ปล่อยให้ผู้ช่วย AI ช่วยเขียนโค้ดดีกว่า แค่คืนเดียว เครื่องมือเล็กๆ ที่ต้องการก็เสร็จเรียบร้อย "โค้ดกาว" ทั้งหมด AI สามารถทำได้เกือบจะสมบูรณ์แบบ อย่างไรก็ตาม ฉันค่อนข้าง保守 มักจะใช้แค่ "ถามตอบอัจฉริยะ" ของ Tongyi Lingma เท่านั้น โค้ดที่ได้มา ฉันมักจะตรวจสอบเองก่อนแล้วจึงคัดลอกและวาง แม้ว่าตัวเลือก "แก้ไขไฟล์" และ "ตัวแทนอัจฉริยะ" อาจมีประสิทธิภาพ更高 แต่ฉันกลัวว่าจะทำให้โค้ดยุ่งเหยิง 反而得不偿失
เมื่อมองไปที่แพลตฟอร์ม Low-code แล้ว ที่จริง门槛การใช้งานก็ไม่ต่ำเลย เช่นเครื่องมืออย่าง n8n แต่ละองค์ประกอบก็ยังมีพารามิเตอร์บางตัวที่ต้องกำหนดค่า จำเป็นต้องเข้าใจ syntax พื้นฐานของ JS ต้องรู้จัก JSON เมื่อเกิดข้อผิดพลาดก็ต้องรู้วิธีแก้ไขปัญหาในสภาพแวดล้อมการติดตั้งด้วย ความรู้พื้นฐานทางคอมพิวเตอร์เหล่านี้ ไม่ว่าคุณจะใช้เครื่องมือหรือแพลตฟอร์มอะไรก็ตาม ที่จริงแล้วก็หนีไม่พ้นที่ต้องเรียนรู้อยู่ดี
那么低代码平台就不能用了吗？ แปลเป็นภาษาไทยคือ: แพลตฟอร์ม Low-code ก็ใช้ไม่ได้แล้วเหรอ?
คำตอบของฉันคือ: ใช้ได้ และยังมีประโยชน์มาก! ประการแรก เมื่อเริ่มต้นใช้งาน แผนภูมิการไหลของงานจะช่วยให้เข้าใจกระบวนการทำงานได้อย่างชัดเจน เป็นการมองเห็นภาพรวมว่าเราจะแบ่งงานใหญ่เป็นส่วนย่อยๆ ได้อย่างไร เครื่องมือเหล่านี้ได้ออกแบบโครงสร้างพื้นฐานไว้เรียบร้อยแล้ว ไม่จำเป็นต้องคิดเองให้ยุ่งยาก โครงสร้างเหล่านี้ยังช่วยจัดลำดับความคิดเวลาทำโค้ดดิ้งได้ดี ประการที่สอง ตัวอย่างจำนวนมากจะช่วยให้ผู้เรียนสามารถออกแบบการประยุกต์ใช้ให้เหมาะกับความต้องการของตัวเอง ลองดูว่ากระบวนการใดบ้างที่สามารถเพิ่มประสิทธิภาพด้วยระบบอัตโนมัติ เมื่อเรามี "ค้อน" อย่าง LLM แล้ว การดูว่าคนอื่นใช้ค้อนตอกอะไรบ้าง ย่อมง่ายกว่าการถือค้อนออกไปหาตะปูเอง
ความคิดเห็นสุดท้าย: ในปัจจุบัน เครื่องมือที่มีประสิทธิภาพสูงสุดยังคงเป็น vscode หรือ pycharm ร่วมกับผู้ช่วย AI เช่น Cline และ Tongyi Lingma ในขณะที่พนักงานที่มีคุณค่ามาที่สุดยังคงเป็นผู้มีความเชี่ยวชาญหลายด้าน ที่มีทั้งประสบการณ์และแนวคิดทางธุรกิจ พร้อมด้วยพื้นฐานความรู้ด้านวิทยาการคอมพิวเตอร์

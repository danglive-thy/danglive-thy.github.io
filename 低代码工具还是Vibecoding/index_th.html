เอชทีเอ็มแอล เครื่องมือ Low-Code หรือ Vibe coding?

เครื่องมือ Low-code หรือ Vibe coding?

ต้นฉบับ ดังไลฟ์ แดนกลิฟ์
วันจันทร์ที่ 5 พฤษภาคม 2568 เวลา 14:32 น. เซี่ยงไฮ้
มนุษย์มักจะชอบการหลีกเลี่ยงความยุ่งยาก ถ้าสามารถสร้างแอปพลิเคชันได้โดยไม่ต้องเขียนโค้ด แค่ลากและดึงก็พอแล้ว ใครจะไปสนใจการเขียนโค้ดล่ะ? แต่แพลตฟอร์มแบบ Low-Code สามารถทำงานได้อย่างสมบูรณ์จริงหรือ? หรือมันเป็นแค่ของเล่นสำหรับผู้เริ่มต้นเท่านั้น?
กลับไปที่ครึ่งปีก่อน: ตอนเริ่มเรียนรู้การพัฒนา ESP32 แรกๆ ไม่อยากเรียนรู้โค้ด micropython จึงหวังว่าจะมีเครื่องมือ low-code ที่สามารถเห็นแบบนั้นได้ทันที แค่ลากและวาง ใส่พารามิเตอร์ไม่กี่ตัวก็สามารถสร้างโปรแกรมได้ แต่ในความเป็นจริงนั้น หลังจากลองใช้เครื่องมือ low-code ทั้งในและต่างประเทศแล้ว กลับพบว่าต้องกลับไปพัฒนาที่ pycharm และ thonny อีกครั้ง ซึ่งมีประสิทธิภาพสูงสุด
ก่อนอื่น ตัวเครื่องมือ Low-code เป็นเหมือนกล่องดำ ตัวอย่างเช่น โค้ดที่ถูกคอมไพล์ออกมา มีความอ่านง่ายต่ำ ทำให้เราไม่สามารถเข้าใจได้ว่าทำไมสิ่งต่าง ๆ ถึงเกิดขึ้นอย่างนั้น เมื่อเกิดปัญหาที่ไม่สามารถตอบสนองตามความต้องการได้ การ debug แทบเป็นไปไม่ได้ และการแก้ปัญหาต้องอาศัยการเดา ตัวอย่างเช่น ฟังก์ชันที่ทำงานได้ดีในสภาพแวดล้อมจำลอง เมื่อถูกนำไปใช้งานจริงกับ SoC กลับไม่สามารถทำงานได้ ทำให้ยากที่จะระบุว่าปัญหาเกิดขึ้นจากที่ใด
ประการต่อไป คือสามารถเรียนรู้ได้เพียงผ่านกรณีศึกษาเท่านั้น แต่จำนวนของกรณีศึกษานั้นอาจน้อยเกินไป สำหรับฟังก์ชันที่ต้องการใช้งาน หากไม่มีกรณีศึกษาที่เกี่ยวข้อง ก็จะต้องไปค้นหาที่ต่าง ๆ ซึ่งกินเวลานานและใช้แรงงานมาก
ดังนั้นหลังจากใช้ไปประมาณ 1-2 สัปดาห์ ก็ตัดสินใจกลับมาใช้ PyCharm อีกครั้ง และใช้ Tongyi Lingma เป็น AI ช่วยเขียนโปรแกรม ซึ่งจริงๆ แล้วมีประสิทธิภาพสูงมาก ที่ผ่านมา Tongyi Lingma ได้รับการปรับปรุงให้เหมาะกับการเขียนโค้ดด้าน IoT ดังนั้นจึงเลือกใช้มันในตอนนั้น
แน่นอน แม้ว่าคุณจะใช้ AI ช่วยเขียนโปรแกรม แต่คุณเองยังต้องมีพื้นฐานการเขียนโปรแกรมอยู่ดี ไม่เช่นนั้นคุณจะไม่สามารถแยกแยะปัญหาในโค้ดที่ AI ให้คุณได้ หลายครั้ง AI ช่วยเขียนโค้ดเพียงแค่ 2 นาที แต่คุณอาจต้องใช้เวลาถึง 2 วันในการแก้ไขปัญหาเดียว
การเขียนโค้ดแบบ Vibe coding มีข้อดีที่เห็นได้อย่างชัดเจนในสองประเด็นหลัก เมื่อเทียบกับเครื่องมือ Low-code ข้อแรกคือการ debug เมื่อเกิดปัญหาคุณสามารถอ่านโค้ดด้วยตนเอง หรือส่งข้อความข้อผิดพลาดไปให้ AI ช่วยแก้ไขอย่างต่อเนื่อง ซึ่งทำให้คุณมีความควบคุมที่สูงขึ้นมาก ข้อที่สองคือไม่ต้องเสียเวลาในการค้นหาอีกต่อไป เพียงแค่บอกความต้องการให้ AI รู้ ก็สามารถสร้างโค้ดได้อย่างต่อเนื่อง แม้แต่ไลบรารีบางอย่างที่เคยต้องค้นหาเอง ก็สามารถเขียนได้ด้วยตนเองแล้ว
เช่น ตัวอย่างเช่น เมื่อหกเดือนก่อนที่ต้องการไดรเวอร์สำหรับแสดงผลบนจอแสดงผลแบบดิจิทัล ไลบรารีบน Github ที่มีอยู่นั้นไม่ค่อยเหมาะสม ไลบรารีที่มีนั้นเป็นเฉพาะสำหรับจอแสดงผลแบบดิจิทัลรุ่นนั้นที่ใช้การเชื่อมต่อแบบ Common Cathode และมีเฉพาะในรูปแบบภาษา C แต่จอแสดงผลที่ฉันใช้เป็นแบบ Common Anode และต้องใช้รูปแบบภาษา micropython ดังนั้นจึงใช้ AI Assistant ช่วยแปลงโค้ดไดรเวอร์ที่เป็นภาษา C ให้เป็นภาษา Python ซึ่งทำได้ง่ายมาก แต่เพียงแค่เปลี่ยนจาก Common Cathode เป็น Common Anode นั้น AI Assistant ยังไม่สามารถเข้าใจได้ จึงต้องใช้เวลาแก้ไขตรรกะด้วยตนเอง หากโครงการทั้งหมดใช้แพลตฟอร์ม Low Code อาจพบกับปัญหาใหญ่ เพราะไม่มีไลบรารีหรือไดรเวอร์ที่ใช้ได้เลย จึงอาจต้องเปลี่ยนฮาร์ดแวร์แทน
การพยายามติดตั้ง LLM บนท้องถิ่นล่าสุด ฉันก็เจอกับปัญหาที่คล้ายกันอีกครั้ง ช่วงแรก ฉันถูกบทความที่แนะนำ dify, coze, n8n บน Xiaohongshu ดึงดูด จึงเริ่มศึกษาผ่าน Bilibili สำหรับ dify ปัญหาเริ่มตั้งแต่การติดตั้งด้วย docker ซึ่งทำให้หงุดหงิดอย่างมาก เนื่องจากมีปัญหาในการติดตั้ง docker ในสภาพแวดล้อม WSL บน windows ของฉัน ดูเหมือนจะไม่สามารถใช้ proxy ที่อยู่ภายใต้ windows ได้ ทำให้ตอนติดตั้ง docker ไม่สามารถเชื่อมต่อกับเซิร์ฟเวอร์ได้ จนในที่สุดก็ต้องยอมแพ้ หลังจากนั้น ทุกคนบอกว่า n8n นั้นพัฒนามากกว่า ดูแล้วก็เป็นเช่นนั้นจริง จึงเปลี่ยนมาลองใช้ n8n แทน
ในตอนแรก ต้องยอมรับว่ามันค่อนข้างสมบูรณ์แบบ น8น มีส่วนประกอบที่ผสานเข้าด้วยกันได้มากกว่า几百 ชิ้น ซึ่งสามารถเชื่อมต่อกับ AI Agent ได้ และการเรียกใช้ LLM ที่ติดตั้งผ่าน ollama บนท้องถิ่นนั้นทำได้ค่อนข้างง่าย ตอนที่ลองใช้งานเบื้องต้นก็ยังโอเค แต่เมื่อเริ่มใช้งานลึกขึ้น... ปัญหาเริ่มเกิดขึ้น: ตอนที่ฉันต้องการอ่านไฟล์ท้องถิ่นในงานโฟลว์จริง มันเริ่มแสดงข้อผิดพลาด莫名其妙 หลังจากเพิ่มส่วนประกอบ "read file" มันก็เริ่มแสดงข้อผิดพลาดเกี่ยวกับ date-fns ฉันส่งข้อความข้อผิดพลาดให้ AI ดู แล้วมันแนะนำให้ฉันติดตั้งเวอร์ชันล่าสุดของ date-fns ทั่วโลก ดี ฉันลองทำตามคำแนะนำ! ดูเหมือนจะดีขึ้น? แล้วเมื่อฉันเพิ่มส่วนประกอบ "extract from file" มันก็ยังคงแสดงข้อผิดพลาดต่อไป ครั้งนี้ไม่มีทิศทางเลย แทบไม่สามารถแก้ไขได้เลย ฉันลองติดตั้งใหม่หลายครั้ง และโหลดโฟลว์ตัวอย่างจากเว็บไซต์อย่างเป็นทางการของมัน แต่ปัญหายังคงเกิดขึ้นเหมือนเดิม
สุดท้ายแล้วก็ต้องยอมรับความจริง และกลับไปใช้ VSCode อย่างขยันขันแข็ง ให้ AI ช่วยในการเขียนโปรแกรมเสียที แค่เพียงหนึ่งคืน ก็สามารถสร้างเครื่องมือเล็กๆ ที่ต้องการได้สำเร็จ ทุกชิ้นส่วนของโค้ดที่ใช้เชื่อมต่อระหว่างส่วนต่างๆ AI ก็สามารถทำได้ใกล้เคียงความสมบูรณ์แบบ อย่างไรก็ตาม ฉันยังคงระมัดระวังอยู่เสมอ โดยทั่วไปจะใช้เพียงฟังก์ชัน “การตอบคำถามอย่างชาญฉลาด” ของ Tongyi Lingma เท่านั้น โค้ดที่ได้รับจากมัน ฉันมักจะตรวจสอบด้วยตนเองก่อนค่อยคัดลอกและวางลงไป แม้ว่าฟังก์ชัน “การแก้ไขไฟล์” และ “ส่วนตัวอัจฉริยะ” อาจมีประสิทธิภาพสูงกว่า แต่ฉันกลัวว่าการใช้พวกมันอาจทำให้โค้ดเกิดความผิดพลาด จนไม่คุ้มค่าในที่สุด
ในทางกลับกัน แพลตฟอร์ม Low-code นั้น แท้จริงแล้วมีข้อจำกัดในเรื่องการใช้งานที่ไม่ได้ต่ำกว่ามากนัก ตัวอย่างเช่นเครื่องมืออย่าง n8n ที่แม้จะเป็นเครื่องมือ Low-code แต่ก็ยังมีการตั้งค่าพารามิเตอร์ในแต่ละคอมโพเนนต์ที่ต้องใช้ความรู้พื้นฐานของภาษา JavaScript และ JSON รวมทั้งต้องเข้าใจวิธีแก้ไขปัญหาที่เกิดขึ้นภายในสภาพแวดล้อมการติดตั้ง ความรู้พื้นฐานเหล่านี้ไม่ว่าจะใช้เครื่องมือหรือแพลตฟอร์มใดๆ ก็ตาม ย่อมหลีกเลี่ยงไม่ได้ที่จะต้องเรียนรู้
ดังนั้นแพลตฟอร์ม Low-Code ไม่สามารถใช้งานได้แล้วหรือ?
คำตอบของฉันคือ: ใช่ได้ ยังคงมีประโยชน์มาก! เหตุผลประการแรกคือเมื่อเริ่มต้น แผนภาพที่แสดงขั้นตอนการทำงานจะช่วยให้เข้าใจการจัดการขั้นตอนการทำงานได้อย่างเข้าใจง่าย สามารถเห็นได้อย่างชัดเจนว่างานใหญ่หนึ่งงานสามารถแบ่งย่อยลงไปสู่งานย่อยๆ ได้อย่างไร และเครื่องมือเหล่านี้ก็ได้สร้างการ抽象ที่ทั่วไปไว้เรียบร้อยแล้ว ไม่จำเป็นต้องพยายามคิดเองอีกต่อไป การabstractเหล่านี้ยังเป็นประโยชน์มากในการเขียนโปรแกรมอย่างมีความรู้สึก (vibe coding) โดยช่วยให้คิดเรียงลำดับความคิดได้ชัดเจนขึ้น เหตุผลประการที่สองคือมีตัวอย่างมากมาย ซึ่งสามารถสร้างแรงบันดาลใจให้ผู้เรียนสามารถคิดค้นและสร้างสถานการณ์ใช้งานที่เหมาะสมกับตนเองได้ รวมทั้งสามารถพิจารณาได้ว่างานใดบ้างที่สามารถเพิ่มประสิทธิภาพได้อย่างมากผ่านการอัตโนมัติ เมื่อมี LLM ที่เป็น "ไขควง" แล้ว ดูว่าคนอื่นเคย "ขันตะปู" อะไรไว้บ้าง ย่อมดีกว่าการที่เราถือไขควงไปหาตะปูเองอยู่ทุกที่นั่นเอง
ความคิดเห็นสุดท้าย: ในปัจจุบันเครื่องมือที่มีประสิทธิภาพสูงสุดยังคงเป็น VSCode หรือ PyCharm ร่วมกับผู้ช่วย AI เช่น Cline หรือ Tongyi Lingma ขณะเดียวกันพนักงานที่มีค่าที่สุดยังคงเป็นบุคคลที่มีประสบการณ์และทักษะทางธุรกิจ รวมถึงมีพื้นฐานความรู้ด้านคอมพิวเตอร์ ซึ่งเป็นคนที่มีความสามารถหลากหลาย
